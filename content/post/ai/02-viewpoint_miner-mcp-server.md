本篇先讲 MCP——把数据系统变成 AI 能调用的工具集。

---

MCP（Model Context Protocol）​ 是 Anthropic 提出的开放协议，核心思路很简单：让 LLM 能"调用工具"。不是让 LLM 写 SQL、调 API，而是把数据能力封装成带类型签名和描述的函数，LLM 根据用户意图自动选择合适的工具并传参。

举个例子。以前查"北方稀土有没有人喊减仓"，我得：切浏览器 → 打开系统 → 搜索股票 → 点进去 → 找信号标签 → 回到编辑器。现在直接跟 WorkBuddy 说一句就行，AI 替我调系统、查数据、汇总结果。
这就是 MCP 要解决的问题。

---

## 架构：从 REST 到 MCP 的思维转变

REST API 和 MCP 工具看起来很像——都是"请求-响应"，都有参数和返回值。但设计哲学完全不同：

| | REST API | MCP 工具 |
|---|---|---|
| **调用者** | 人类（通过前端 JS） | LLM（自动决策） |
| **参数验证** | 靠前端表单 + 后端校验 | Zod schema 同时服务 LLM 和 runtime |
| **返回值** | 自由格式 JSON | 结构化 `{ content: [{ type: 'text', text: '...' }] }` |
| **发现机制** | Swagger/手写文档 | LLM 读取工具名 + 描述 + 参数 schema 自动理解 |
| **错误处理** | HTTP 状态码 + error body | throw Error → MCP SDK 转协议错误 |
| **组合调用** | 前端手动串联 | LLM 自主编排调用顺序 |

关键区别在于**调用者**。REST API 是为人类前端设计的，Endpoint 命名、参数结构、返回格式都围绕"前端怎么渲染"来定。MCP 工具是为 LLM 设计的，每个工具必须自描述到**不看源码就能用对**的程度。

说白了，REST API 是给人看的，MCP 工具是给 AI 看的。给 AI 看的东西，你得把说明书写在接口里。

## 工具分层：27 个工具怎么组织

下面这张图展示了完整的架构：

![MCP 工具分层架构](/images/viewpoint_miner/mcp-tool-hierarchy.svg)

整体分为 6 层，从底向上依次是：

| 层级 | 名称 | 定位 |
|------|------|------|
| Layer 0 | 数据同步与提取 | 原子操作层，负责原始数据获取 |
| Layer 1 | 基础查询 | 对已同步数据进行简单查询 |
| Layer 2 | 交易信号 | 聚焦交易行为与信号 |
| Layer 3 | 聚合分析 | 统计、分析、整合 |
| Layer 4 | 语义搜索 | 自然语言查询与向量检索 |
| Layer 5 | 自动化通知 | 面向用户的交互与通知 |

分层不是架构洁癖，而是 LLM 工具选择的自然规律：**底层工具是原子操作，上层工具是多步编排**。LLM 会优先选择高层的、语义匹配的工具；当高层工具不存在时，才会退到低层组合调用。

---

## 各层详解

### Layer 0：数据同步与提取（原子操作层）

这是整个架构的最底层，负责从外部数据源获取原始数据。这一层的工具都是**原子操作**，不可再拆分。

- **同步作者观点**：从财经社区抓取特定作者的最新观点
- **同步股票讨论**：同步某只股票的论坛讨论帖
- **批量提取交易动作**：从文本中批量识别买卖操作

这一层的特点是：**执行成本高**（涉及网络请求、数据清洗），因此上层工具应该尽量避免频繁调用。

---

### Layer 1：基础查询

对已经同步到本地的数据进行简单查询，不涉及复杂的计算或推理。

- **获取股票摘要**：返回某只股票的基本面信息
- **搜索股票**：根据关键词模糊匹配股票代码
- **获取股票列表**：返回当前关注的所有股票

这一层是上层分析的**数据底座**，所有聚合操作最终都会落到这里。

---

### Layer 2：交易信号

专门处理与交易行为相关的数据和信号。

- **搜索交易信号**：根据条件筛选特定的交易信号
- **获取最新交易信号**：返回最近产生的交易信号
- **获取信号列表**：按时间线展示所有信号

这一层开始引入**业务语义**，不再是简单的 CRUD。

---

### Layer 3：聚合分析

对底层数据进行统计、分析和整合，产出有洞察力的结论。

- **汇总信号偏向统计**：统计一段时间内的多空信号比例
- **构建股票上下文**：将多个维度的数据打包成一个 RAG 上下文包
- **生成情绪热力图**：可视化展示市场情绪分布

这一层的工具通常**返回的是分析结果而非原始数据**，是 LLM 可以直接使用的“半成品”。

---

### Layer 4：语义搜索

支持自然语言查询和向量检索，让 LLM 能够像聊天一样搜索数据。

- **自然语言搜索观点**：用大白话搜索观点，例如“最近大家对茅台怎么看”
- **RAG上下文构建**：为检索增强生成准备上下文材料
- **向量索引**：将文本转为向量并建立索引

这一层引入了**语义理解**能力，大幅降低了 LLM 的使用门槛。

---

### Layer 5：自动化通知

面向用户的最终输出层，将分析结果转化为可感知的消息或报告。

- **每日摘要**：生成当天的市场观点摘要
- **信号变化检测**：监控信号变化并及时告警
- **多渠道推送**：通过邮件、微信、钉钉等渠道发送通知

这一层是**用户可见的终点**，也是整个架构的价值出口。

---

## LLM 如何选择工具？

理解了分层结构之后，最关键的问题来了：**LLM 到底是怎么选工具的？**

答案是：**优先选上层，找不到再往下拆。**

下面这张图展示了 LLM 的决策路径：

![LLM 决策路径](/images/viewpoint_miner/llm-select-tool.png)

具体来说，LLM 的决策逻辑是这样的：

1. **判断用户意图**：用户想要什么？是摘要、分析、搜索还是基础查询？
2. **匹配最高层工具**：优先尝试用 Layer 5 或 Layer 4 的工具来满足需求
3. **逐层降级**：如果高层工具无法满足，就拆解成下一层工具的调用组合
4. **组合执行**：最坏情况下，LLM 需要从 Layer 0 开始组装一条完整的调用链

举个例子，当用户问“这只股票多空比例怎么样”时：

- LLM 首先匹配到 Layer 3 的 `汇总信号偏向统计`
- 这个工具发现自己需要先获取信号数据，于是调用 Layer 2 的 `搜索交易信号`
- `搜索交易信号` 发现数据还没同步，于是调用 Layer 0 的 `批量提取交易动作`

整个过程就像**剥洋葱**，从外层一层层向内展开，直到找到可以执行的原子操作为止。

## 与 WorkBuddy 的集成

MCP Server 本身只是一个 JSON-RPC 进程，由于我 vibecoding 用的是 Workbuddy，所以这里介绍怎么集成到 WorkBuddy里。

![MCP 协议交互流程](/images/viewpoint_miner/mcp-protocol-flow.svg)

WorkBuddy 通过 `~/.workbuddy/mcp.json` 发现它，配置本身很简单：

```json
{
  "mcpServers": {
    "stock-viewpoints": {
      "command": "node",
      "args": ["/Users/prodeer/Documents/projects/stock-viewpoints/mcp-server.js"],
      "cwd": "/Users/prodeer/Documents/projects/stock-viewpoints"
    }
  }
}
```

配置好之后，WorkBuddy 启动时自动拉起 MCP 进程，读取全部工具列表（名字 + 描述 + 参数 schema），注入到 LLM 的可用工具集中。

## 工具数量：27 是多还是少？

一个常见问题是：27 个工具是不是太多了？会不会让 LLM 困惑？

实际上，工具多不是问题，**工具之间的语义重叠**才是问题。如果三个工具都能做"查股票"，LLM 就不知道该选哪个。这 27 个工具的设计遵循两个原则避免了这个问题：

1. **每个工具有唯一的语义定位。** `search_stocks` 是模糊搜索，`get_stock_summary` 是精确汇总，`get_stock_viewpoints` 是观点列表——三个工具操作不同的抽象层级，不会混淆。

2. **参数 schema 天然区分。** 即使两个工具名字相似，不同的必填参数会让 LLM 做出正确选择。用户说"分析北方稀土"时，LLM 知道自己需要先拿到 stock_id，所以必须先调 `search_stocks`，再调 `summarize_stock_signals`。

实际经验是：LLM 选工具的准确率远高于人类的直觉预期。只要工具描述清晰、参数类型明确、语义无重叠，27 个工具不会造成选择困难。

---

## 小结

这篇文章从架构层面拆解了 MCP Server 的设计：

1. **分层解耦**：每一层职责明确，上层依赖下层但不越级。新增一个工具只需要在对应层级添加，不影响其他层。
2. **LLM 优先策略**：让 LLM 优先调用高层聚合工具，减少调用链长度，提升响应速度和准确性。
3. **RAG 友好**：多层架构天然支持检索增强生成场景：Layer 4 负责检索，Layer 3 负责整合，Layer 5 负责输出。


但 MCP 工具只是"能力暴露"。真正的魔法在于**让 AI 根据用户的一句话自动编排工具链路**——这就是下一篇的主题：Skills 封装。
