## 怎么从爱股票抓数据

![数据管道架构图](/images/trade_copier/data-pipeline.svg)

### 两个 API 端点

爱股票平台提供了两个关键接口：

| API | 方法 | 用途 |
|---|---|---|
| `live_center_v3.php` | GET | 按时间倒序拉取作者直播间消息流 |
| `liveCenterSearch` | POST | 在作者全量历史中搜索含特定股票关键词的观点 |

第一个用于**日常增量同步**（每天拉新观点），第二个用于**补全股票覆盖**（新关注的股票回头搜历史观点）。

两个 API 都不需要 Cookie，只需要参数填对——最关键的是 `id` 参数必须是**直播间 ID**（`l_id`），不是用户 ID（`u_id`）。这个坑是我从作者主页 HTML 里正则提取 `"l_id":"4277"` 才填上的。

### 数据模型

核心三张表：

```
authors ──┐
          │ 1:N
          ▼
      viewpoints ──┐
                   │ M:N
                   ▼
          viewpoint_stocks ── stock_id ──▶ stocks
```

- **authors**：存储爱股票作者信息，关键字段 `aigupiao_lid`（直播间 ID）
- **stocks**：股票池，`code` 带 `sz`/`sh`/`hk` 前缀，`is_starred` 标记关注
- **viewpoints**：观点原文（`content` 是纯文本，`raw_content` 保留原始 HTML），`rec_time` 是 API 返回的秒级时间戳，同时也是去重键

还有一张派生表 **trade_actions**（后面会详细讲）。

### 数据清洗

从 API 拿到的每条消息是一坨 JSON，经过 `parseMessage()` 处理：

1. **图片处理**：`image` 字段是字符串数组，join 存储
2. **股票解析**：`stock_info` 数组 `[{code:"sz300014", name:"亿纬锂能"}]`，剥离前缀得到纯代码，取 market 判断 A/HK/US
3. **HTML 清洗**：正则去标签，`&nbsp;` → 空格，`&lt;` → `<`
4. **转发处理**：`forward_info` 数组过滤空元素后 JSON 存储
5. **时间转换**：秒级时间戳 → ISO 8601
> rec_time是 API 返回的秒级 Unix 时间戳，解析时乘 1000 转成 ISO 8601（2026-05-18T10:00:00+08:00），方便存储和排序。

### 增量同步的翻页逻辑

直接贴代码：

```javascript
let before = 0;
let page = 0;
while (page < MAX_PAGES) {
  const res = await axios.get(API, { params: { id: l_id, before, source: 'pc', act: 'msg' } });
  const list = res.data.msg_list || [];
  if (list.length === 0) break;
  
  for (const item of list) {
    if (item.rec_time <= lastRecTime) return; // 到边界了，停止
    insertViewpoint(parseMessage(item));
  }
  
  before = list[list.length - 1].rec_time; // 用本页最后一条的时间戳翻页
  page++;
  await sleep(300); // 防反爬
}
```

`before` 游标配合 `rec_time` 边界检查，保证不重不漏。首次同步从 2026-01-01 开始，之后只拉增量。

## 信号提取引擎：规则驱动的中文 NLU

### 为什么是规则引擎而不是 LLM？

2026 年了，大家都在聊 Agent、聊 MCP、聊大模型。但面对"从投资观点中提取交易信号"这个任务，我选了纯规则：

1. **精确性**：投资领域的动作词高度集中——建仓、加仓、减仓、清仓、止盈、止损——词典就二十几个，不需要模型的"理解"
2. **价格提取是文本匹配问题**：数字就长那样，regex + 启发式策略比 LLM 更可靠
3. **零依赖**：不需要 API key，不需要 GPU，不需要 Python 环境。一个 600 行的 .js 文件搞定
4. **可调试**：每步策略都是白盒，出问题了能精确知道哪个规则匹配错了，而不是"模型幻觉"
5. **成本**：2000+ 条观点，每秒几百条，LLM API 调用费用完全没必要

**当问题域足够窄、模式足够清晰时，规则引擎是最优解。** 这不是反 AI，而是务实。

### 动作关键词体系

引擎核心是一张动作分类表：

```javascript
const ACTION_RULES = [
  // 买入类
  { action: '建仓', keywords: ['建仓', '打底仓', '开仓', '首仓', '初始仓'], type: 'buy' },
  { action: '加仓', keywords: ['加仓', '补仓'], type: 'buy' },
  { action: '买回', keywords: ['买回', '接回', '抄底', '接回来', '买回来'], type: 'buy' },

  // 卖出类
  { action: '减仓', keywords: ['减仓', '走了', '跑了', '先走', '减一半', '先跑一半'], type: 'sell' },
  { action: '踢出', keywords: ['踢出', '踢出去', '出掉'], type: 'sell' },
  { action: '卖出', keywords: ['卖出', '抛出'], type: 'sell' },
  { action: '清仓', keywords: ['清仓', '格局', '全出', '全卖'], type: 'sell' },
  { action: '止盈', keywords: ['止盈', '获利了结', '锁定利润'], type: 'sell' },
  { action: '止损', keywords: ['止损', '割肉', '认赔'], type: 'sell' },

  // 观察类（大幅降权）
  { action: '观察', keywords: ['关注'], type: 'watch' },
];
```

### 价格提取：五层优先级策略

价格提取是整个引擎最复杂的部分。不是简单的"找到数字就行"——要考虑价格出现在动作词的前面还是后面、是单价还是区间、是精确的还是"附近"。

引擎按优先级从高到低尝试：

#### 策略 1：后方紧跟数字（最高优先级）

```javascript
// "买回395" → 价格 395
const afterText = content.substring(keywordEnd, keywordEnd + 30);
const match = afterText.match(/^\s*(\d+\.?\d*)/);
```

如果数字后面紧跟 `-`/`~`/`至`/`到` 和另一个数字，识别为区间：

```javascript
// "清仓51.8-52.5" → price_low: 51.8, price_high: 52.5
```

如果数字后面跟「附近/左右/上下/一带」，标记为近似价格。

#### 策略 2：前方紧邻区间

```javascript
// "7.97-8.15清仓" → price_low: 7.97, price_high: 8.15
// 区间末尾到关键词不超过 8 个字符
```

#### 策略 2.5：前方近似价格

```javascript
// "26附近可以走了" → 26
// 允许中间有"可以/能/就/可"等短词
```

这是专门为「X 附近可以走了/跑了」这种模式设计的。如果不做近似匹配，"26 附近可以走了"会匹配不到价格。

#### 策略 3：前方紧邻单价

```javascript
// "26加仓" → 26
// 但要排坑：不能吃区间尾巴——"7.97-8.15清仓"中的 8.15 不是单价
```

#### 策略 4 & 5：后方区间 + 远距离修饰

```javascript
// 后方区间："清仓51.8-52.5"
// 远距离："23以上减仓"、"30上方加仓"、"36下方减仓"
```

这五层策略的排列顺序本身就是一种启发式编码：**越靠近动作词的价格越可信，前方的比后方的更可能是操作价位。**

### 置信度评分：七因子调整

提取到动作和价格只是第一步。真正区分"真实信号"和"文字噪音"的是置信度系统。

基线从 **0.5** 开始，通过以下因子调整：

| 条件 | 调整 | 设计意图 |
|---|---|---|
| 提取到有效价格 | **+0.3** | 有价格的信号意图明确得多 |
| 精确动作词 | **+0.1** | "建仓"比"关注"更像操作 |
| 句首位置 | **+0.05** | 独立分句的开头更可能是新操作 |
| 观察类动作 | **-0.3** | "关注"噪音太大 |
| 观察但有价格 | **+0.2** | "关注 26"比"关注"略可信 |
| 否定语境 | **-0.4** | "不要加仓"不是信号 |
| 泛化后缀 | **-0.4** | "加仓计划/策略/节奏"只是讨论 |
| 反问句 | **-0.4** | "要不要加减？"不是操作 |
| 完成时态「了」 | **-0.4** | "加仓了"——已经做了，不是现在要做的 |
| 完成时态「的」 | **-0.5** | "接回的"、"买回的"——过去完成式 |
| 回顾性语境 | **-0.5** | "之前建仓"、"当时做到 65"——历史叙述 |
| 条件性语境 | **-0.5** | "如果跌到 26 可以加仓"——假设，不是操作 |

最后裁剪到 [0, 1]，低于 **0.5** 的直接丢弃。

### 条件语境 vs 推荐指令：一个微妙的区分

条件性语境的处理有一个细节值得单独说。以下两句都包含"可以"：

> A. "如果跌到 26 可以加仓" → 条件是前面 30 字内的"如果"  
> B. "可以打底仓 26" → 推荐指令，不是条件

引擎的处理逻辑：

- 动作词**前面**有 `如果/若是/要是` → 条件语境，**-0.5**  
- 动作词**后面**紧跟 `可以/能` → 条件语境，**-0.5**  
- 但"可以"在动作词**前面**时（如"可以打底仓"）→ 不是条件，是建议，不降权

这个区分看似琐碎，实际上反映了中文语序的语义功能：**"可以 + V"是祈使/建议，"V + 可以"是条件从句。**

### 买卖归一化：当一条观点同时喊买和喊卖

投资观点经常出现"先出再进"或"先买再卖"的描述：

> "26 附近先出，22-23 再接回来。"

引擎可能同时提取到「减仓 @26」和「买回 @22-23」。这时 `normalizeBuySell()` 登场：

```javascript
function normalizeBuySell(results) {
  // 有"买回" → 归一为"加仓"，丢弃所有卖出类
  //   （因为买回意味着当前仓位低于目标，需要加仓）
  if (buyBack) return [{ ...buyBack, action: '加仓' }];
  
  // 有"踢出" → 归一为"减仓"，丢弃所有买入类
  if (kickOut) return [{ ...kickOut, action: '减仓' }];
  
  // 否则保持原样
  return results;
}
```

优先级：**买回 > 踢出**。逻辑是：如果既说了要卖又说要接回来，整体倾向是波段操作——当前是卖出时机，但长期看多。

### 目标价提取：区分"目标"和"卖出触发价"

除了即时操作价格，投资观点里经常出现目标价：

> "中线目标 35-36，做到 30 可以先减一半。"

这里的 35-36 是目标价，30 是触发价。引擎通过两套模式分别处理：

**目标价模式**（优先级 2，更可信）：
- `目标 X-Y`、`目标价 X`、`中线目标 X-Y`
- `至少做到 X`

**$做到$模式**（优先级 1，需排除干扰）：
- `做到 X-Y`（区间）
- `做到 X`（单价）——但要排除「做到 X 可以走了」这种卖出触发价

排除规则的关键：当"做到 X"后面 15 字内出现「走了/跑了/出了/减仓」这些卖出意图词时，X 不是目标价，而是操作触发价。

### 执行流程

把以上所有模块穿起来，提取交易动作的执行流程是这样：

```
输入：一条观点文本

├─ 遍历动作关键词表，全文搜索每个关键词
│   ├─ 名词化检测："所有加仓"、"这些加仓" → 跳过
│   ├─ 重叠检测：已被匹配的文本片段不再使用
│   ├─ 提取价格：五层策略依次尝试
│   ├─ 提取仓位比例：匹配"一半"、"底仓"、"全仓"等
│   ├─ 置信度计算：七因子加减，低于 0.5 丢弃
│   └─ 
│
├─ 去重：按动作类型加价格区间合并重复项
│
├─ 买卖归一化：
│   ├─ 有"买回" → 归一为"加仓"，丢弃卖出类
│   └─ 有"踢出" → 归一为"减仓"，丢弃买入类
│
└─ 提取目标价：
    ├─ 匹配六种目标价模式
    ├─ 排除回顾性描述、卖出触发价、非价格用法
    └─ 优先附加到买入类动作

输出：[{ action, price_low, price_high, target_price_low, target_price_high, confidence }]。
```

600 行代码，零依赖，处理 2000+ 条观点耗时不到 1 秒。

### 引擎设计的几个感悟

做完这个引擎后，有几个体会值得记下来：

**1. 规则的顺序就是优先级。** 价格提取的五层策略不是并列的，而是 fallthrough 的优先级链。前面的策略匹配到了，后面的就不执行。这种"排队尝试"的模式比一个大正则灵活得多。

**2. 消极信号比积极信号更重要。** 大部分规则引擎倾向于"找到什么算什么"，但在这个场景下，**排除假阳性**（回顾性、条件性、反问句）比**提高召回率**更关键。一条假的操作信号可能导致推送骚扰，而漏掉一条真实信号只是少了一条提醒。所以置信度系统里，扣分的因子比加分的因子多得多。

**3. 口语化表达很难缠，但不是无解。** ​ 标准词就那么几个，但口语词层出不穷————“走了”、“跑了”、“溜了”、“先撤了”，今天加完明天又来新的。解决办法不是堆关键词，而是做三层防御：一是映射表与核心逻辑解耦，新增口语词不改代码；二是模糊匹配，容忍插入语和重复；三是置信度加权，口语词自动打折，避免低质量信号污染结果。更治本的做法是从历史数据中挖掘高频搭配，让规则引擎具备自我进化能力。
