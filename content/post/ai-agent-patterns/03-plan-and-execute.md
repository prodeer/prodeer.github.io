把一个旧代码库迁移到新框架，通常不能靠“看到什么就改什么”。需要先识别模块和依赖，再安排改造顺序、测试和切换方案。执行中又可能发现隐藏依赖，需要调整原计划。

Plan-and-Execute 模式用一个显式计划连接目标与执行，让 Agent 在局部动作之外保留全局结构。

<!--more-->

## 模式意图

> 先将复杂目标分解为可执行步骤，再由执行器逐步完成；当关键假设变化时，更新未完成的计划。

它解决的不是“下一步能不能调用工具”，而是“一个长任务怎样避免遗漏、怎样表达依赖、怎样知道整体进度”。

![AI Agent Plan-And-Execute 设计模式](/images/ai/ai-agent-plan-and-execute-pattern.png)

## 问题场景：迁移旧代码库

假设目标是把一个旧 Web 服务迁移到新框架，同时保持接口兼容。任务至少包含：

- 识别入口、依赖和外部契约；
- 迁移公共中间件；
- 逐个迁移业务模块；
- 运行回归测试；
- 制定灰度切换和回滚方案。

这些步骤存在依赖，可以部分并行，也可能在执行中新增工作。纯 ReAct 容易聚焦最近发现的错误，计划则持续提醒 Agent 还有哪些目标没有覆盖。

## 基本结构

{{< mermaid >}}
flowchart LR
    G[Goal] --> P[Planner]
    P --> PS[(Plan)]
    PS --> E[Executor]
    E --> A[Artifact / Observation]
    A --> V[Step Validator]
    V -->|通过| PS
    V -->|关键变化| P
    PS -->|全部完成| O[Outcome]
{{< /mermaid >}}

主要参与者包括：

- **Goal**：任务目标和约束；
- **Planner**：生成或修改计划；
- **Plan**：步骤、依赖、状态和完成标准；
- **Executor**：完成一个就绪步骤；
- **Validator**：判断步骤结果是否满足要求。

Plan 是这个模式最有辨识度的中间对象。它既不是自然语言待办列表，也不是最终答案，而是运行过程的一部分。

## 运行过程

{{< mermaid >}}
flowchart LR
    A[分析目标] --> B[生成计划 v1]
    B --> C[选择就绪步骤]
    C --> D[执行并验证产物]
    D --> E{关键假设改变?}
    E -->|否| F{还有未完成步骤?}
    E -->|是| G[生成计划 v2]
    G --> C
    F -->|是| C
    F -->|否| H[验证整体目标]
{{< /mermaid >}}

一个最小计划可以只保留这些信息：

```python
plan = planner.create(goal)
while step := plan.next_ready():
    result = executor.run(step)
    plan.record(step, result)
    if result.invalidates_assumption:
        plan = planner.revise(plan, result)
return validate_goal(plan.artifacts)
```

执行器内部可以是普通函数、固定 Workflow，也可以使用 ReAct。Plan-and-Execute 不规定每个步骤必须怎样完成。

## 核心特点

### 计划是一种显式状态

没有 Plan 时，全局目标可能只存在于上下文文本里。显式计划使进度、依赖和遗漏变得可检查，也让暂停和恢复更容易。

### 规划和执行关注不同尺度

Planner 关注全局覆盖和步骤关系，Executor 关注当前步骤。职责分开不代表必须使用两个模型，只代表两种决策不应混成一次自由生成。

### 计划允许变化，但不应随意漂移

Replan 是这个模式的重要部分。新事实改变关键假设时应更新计划；普通步骤成功不需要每次重新规划。否则计划会失去稳定全局目标的价值。

### 步骤完成不等于目标完成

所有步骤被标记为完成，仍可能漏掉整体约束。模式通常需要步骤级验证和目标级验证两个层次。

## 优点与代价

{{< mermaid >}}
flowchart TB
    P[显式计划]
    P --> B1[全局可见]
    P --> B2[依赖与并行清晰]
    P --> B3[便于暂停和恢复]
    P --> C1[规划本身有成本]
    P --> C2[计划可能过时]
    P --> C3[错误分解会影响后续]
{{< /mermaid >}}

Plan-and-Execute 提高长任务的结构性，却增加规划调用、状态同步和 Replan 策略。计划越细，管理成本越高；计划越粗，又越难指导执行。

## 适用与不适用场景

适合使用：

- 目标可以分解为多个相对独立的步骤；
- 步骤之间存在依赖、并行或阶段性产物；
- 任务较长，需要追踪进度或中断恢复；
- 执行中可能出现需要调整全局路线的新事实。

不适合使用：

- 一两步即可完成的简单任务；
- 路径完全由每次环境观察决定，提前规划价值很低；
- 流程已经固定且稳定，普通 Workflow 更清晰。

## 与相关模式的区别

| 模式 | 关注点 |
|---|---|
| ReAct | 当前观察之后做什么 |
| Plan-and-Execute | 长目标由哪些步骤组成 |
| Reflection | 当前结果是否需要改进 |
| Multi-Agent | 步骤由哪些角色承担 |

Plan 可以调用这些模式，但不要求同时使用。一个确定性执行器同样可以完成 Plan 中的全部步骤。

## 模式小结

| 维度 | Plan-and-Execute |
|---|---|
| 意图 | 把长目标分解成可管理步骤 |
| 识别信号 | 存在显式、可更新的 Plan |
| 主要优点 | 全局覆盖、依赖清晰、进度可见 |
| 主要代价 | 规划成本、计划过时与状态同步 |
| 使用原则 | 计划只细化到足以指导执行的程度 |
