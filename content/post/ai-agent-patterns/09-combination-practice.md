前面八篇分别介绍了 Tools、ReAct、Plan-and-Execute、Reflection、Memory、RAG、Multi-Agent 和 Autonomous Loop。虽然它们都是常见模式，但这不代表需要全部组合到一个 Agent 里。

实际的设计思路应该反过来——先分析任务中的不确定性和运行约束，再选择能解决具体问题的模式。模式用得越少，系统通常越容易理解、测试和维护。

## 四类任务的模式选择

这篇通过四个典型问题场景，展示如何从任务本身出发做出合理选择。

### 问题一：员工问“出差住宿标准是多少”，答案在公司制度文件里

制度文件是内部的、会动态更新，每次回答都需要引用最新版本。

**最小方案：RAG**

只需要一次检索加一次生成，就能回答问题。

**什么时候加 Tools：** 如果需要按部门过滤文档（比如销售部和研发部标准不同），可以增加一个权限查询 Tool。

**不需要：** ReAct（不需要多步推理）、Plan（不需要提前规划步骤）、Multi-Agent（不需要分工）。

### 问题二：API 挂了，需要查日志、做健康检查，每一步取决于上一步的结果

检查结果会告诉你要不要查数据库、要不要重启服务，下一步永远由上一步决定。

**最小方案：Tools + ReAct**

Tools 负责执行检查动作（查日志、调接口），ReAct 负责根据返回结果决定下一步做什么。

**什么时候替换：** 如果排查步骤完全固定（比如永远是“查健康 → 查日志 → 查数据库”），可以用 Workflow 替代 ReAct，减少 LLM 调用的开销。

**什么时候加：** 如果故障需要持续数小时、等待服务恢复事件，再考虑加入 Autonomous Loop。

**不需要：** 不要一开始就把排查拆成多个 Agent——一个人能看完日志，就不需要三个人开会。

### 问题三：需要把一套旧代码迁移到新架构，涉及依赖识别、分阶段改造、测试验证

迁移不是一步到位的，需要先分析依赖，再分批改造，每做完一部分就测试，测试中可能发现新的问题。

**最小方案：Plan-and-Execute**

先生成一个分阶段的迁移计划，再按计划逐步执行。

**什么时候加 ReAct：** 某个步骤内遇到未知的代码结构，需要临时探索和判断时，可以在局部引入 ReAct。

**什么时候加 Reflection：** 生成迁移方案后，需要自动审查方案质量或测试结果时，可以增加 Reflection。

**不需要：** Plan 不等于必须用 Multi-Agent。一个人能完成的迁移，不要让三个人抢同一个文件。

### 问题四：系统需要跨进程持续跟踪一个故障目标，定期读取指标、等待人工审批、恢复后收尾

这不是几分钟能解决的事。可能需要跑几个小时甚至几天，中间要等人审批、等指标恢复。

**最小方案：Autonomous Loop + Run State**

Autonomous Loop 负责管理运行生命周期（启动、暂停、继续、终止），Run State 负责记录当前进度和状态。

**可以带进来的模式：** 执行检查时需要 Tools，判断诊断方向时需要 ReAct。但这些是因为长期运行的需要才被“带进来”的，不是 Autonomous Loop 默认依赖它们。

**不需要：** 不要把排查过程本身做成 Multi-Agent。一个值班工程师能盯住的告警，不需要拉一个团队来轮班。

## 八种模式的选择矩阵

| 任务信号 | 先考虑什么，以及何时追加 |
|---|---|
| 需要实时数据或业务动作 | 先用 Tools；下一步依赖结果时再加 ReAct |
| 路径依赖环境观察 | 先用 ReAct；需要动作时配合 Tools |
| 多步骤、有依赖和阶段产物 | 先用 Plan-and-Execute；步骤质量不稳时增加 Reflection |
| 结果可评价且需要改进 | 先用 Reflection；评价需要外部材料时增加 RAG 或 Tools |
| 跨任务延续偏好或经历 | 先用 Memory；不要把一次性状态误存为长期记忆 |
| 需要私有、更新、可引用知识 | 先用 RAG；只有需要执行动作时才增加 Tools |
| 存在真实职责或权限隔离 | 先考虑 Multi-Agent；没有边界时保留单 Agent |
| 跨进程持续推进目标 | 先用 Autonomous Loop；按步骤需要再加入 Tools、Plan 或 ReAct |

表格中的“追加”表示可能的协作关系，不表示模式之间存在强制依赖。短任务通常不需要 Plan 或 Autonomous Loop，一次性任务通常不需要 Memory，单一职责通常不需要 Multi-Agent。

## 模式之间怎样组合才有意义

这张图表达的是可能的数据关系，不是推荐的固定装配顺序。组合只有在职责关系真实存在时才有价值：

- Plan 需要把长目标变成步骤，Tools 只提供动作，不会自动替代规划；
- Reflection 需要评价材料，RAG 可以提供证据，但没有 RAG 也可以反思格式或测试；
- Multi-Agent 可以承担不同步骤，但不能替代目标和完成标准；
- Autonomous Loop 管理运行生命周期，不代表每个步骤都必须动态决策。

## 过度设计的三个信号

### 模式解决的问题并不存在

为了“让 Agent 更智能”加入 Memory，却没有跨任务需求；为了“更专业”加入 Multi-Agent，却没有权限或上下文边界。

### 系统无法说明模式的收益

增加一个模式后，只能说调用次数更多、架构更复杂，却无法用完成率、质量、延迟或可维护性证明改善。

### 失败责任变得模糊

当 Planner、Researcher、Reviewer 都可以修改结果，却没有明确产物和所有者时，Multi-Agent 只是把一个难题复制成多个难题。

## 系列总结

这八种模式可以被压缩成八个问题：

| 问题 | 模式 |
|---|---|
| Agent 能做什么 | Tools |
| 现在下一步做什么 | ReAct |
| 长目标有哪些步骤 | Plan-and-Execute |
| 当前结果够不够好 | Reflection |
| 过去还应该知道什么 | Memory |
| 外部知识和证据在哪里 | RAG |
| 谁来承担哪类职责 | Multi-Agent |
| 这次运行是否继续 | Autonomous Loop |

一个好的 Agent 设计，不是让八个模式同时出现，而是只回答当前任务真正提出的问题。学习设计模式的最终成果，也不是拥有更多组件，而是能够解释每个组件为什么存在，以及什么时候可以删掉它。
