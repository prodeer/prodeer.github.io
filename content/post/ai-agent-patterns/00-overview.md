学习传统设计模式时，我们总会追问四个问题：**它解决什么痛点？有哪些角色参与？结构如何组织？引入后又会付出怎样的代价？​**

学习 AI Agent 设计模式亦然。这个系列将聚焦八种常见模式，我们的目标不是用它们拼凑出某种“标准答案”，而是先看清它们——理解其边界与权衡，再谈运用。

<!--more-->

## 这些模式和 GoF 设计模式不完全一样

GoF 模式主要讨论面向对象软件中的可复用结构。Agent 领域还在快速发展，“Agent 设计模式”目前更像一组被反复采用的架构和控制方法。

它们的抽象层次并不完全相同：

- Tools、Memory、RAG 主要扩展 Agent 可以使用的能力和上下文；
- ReAct、Plan-and-Execute、Reflection、Autonomous Loop 主要组织运行过程；
- Multi-Agent 主要改变职责和协作结构。

![AI Agent 设计模式总览](/images/ai/ai-agent-design-patterns-overview.png)

因此，这八种模式不是八类互斥的 Agent，也不是一张实现清单。一个知识库问答可能只使用 RAG；一个故障排查助手可能使用 Tools + ReAct；一个固定文档处理流程可能一种都不需要。

## 八种模式分别在改变什么

| 模式 | 核心意图 | 最鲜明的特点 |
|---|---|---|
| Tools | 让模型调用外部能力 | 模型提出动作，系统掌握执行权 |
| ReAct | 根据观察决定下一步 | 行动与环境反馈交替发生 |
| Plan-and-Execute | 分离全局规划与局部执行 | 计划成为可检查的中间结构 |
| Reflection | 根据评价反馈修订结果 | 生成与评价形成反馈闭环 |
| Memory | 让历史影响未来行为 | 信息跨轮次或跨任务延续 |
| RAG | 在生成前取得外部知识 | 回答建立在检索上下文上 |
| Multi-Agent | 在多个角色间分配工作 | 职责、上下文或权限被隔离 |
| Autonomous Loop | 持续推进一个长期目标 | 运行可以循环、暂停、恢复和终止 |

“谁决定下一步”“信息从哪里来”“完成由谁判断”仍然是识别模式的主要线索。

## 从四个问题识别一个模式

学习每个模式时，可以固定观察四个维度，快速拆解任何模式的结构与意图。

![AI Agent 设计模式识别](/images/ai/identify-agent-pattern-four-questions-landscape.png)


### 它改变了谁的职责

每个模式都在重新分配责任：

- **Tools** 把“想做什么”和“怎么去做”分给两个角色
- **Plan-and-Execute** 让“规划者”和“执行者”各司其职
- **Reflection** 把“写答案的人”和“挑毛病的人”变成两个人
- **Multi-Agent** 干脆把不同专业领域交给不同的智能体

### 它增加了什么状态

模式引入新能力的同时，也在系统中留下了痕迹：

- **ReAct** 记下每一步的观察
- **Plan** 维护待办步骤清单
- **Memory** 跨越多个任务保留上下文
- **Autonomous Loop** 跟踪整个运行周期的状态

记忆越多，系统越聪明，但也越容易“记错”或“矛盾”。

### 它引入了哪种反馈

循环人人都会写，但循环里跑的是什么信号，才决定了模式的身份。

- **环境反馈** → ReAct：下一步该做什么
- **质量反馈** → Reflection：当前答案够不够好
- **历史反馈** → Memory：过去经验如何帮到未来
- **目标进度反馈** → Autonomous Loop：该继续还是停下

看到 `while` 循环不要急着归类——先问一句：**谁在说话？说了什么？**

### 它增加了什么代价

没有免费的午餐，也没有免费的模式：

- **Tools** 要管理权限和副作用
- **Memory** 要处理过期信息和隐私
- **Multi-Agent** 要为沟通买单
- **Autonomous Loop** 要操心什么时候该喊停

## 模式不是越多越好

可以把 Agent 系统看成一条复杂度阶梯：

只有当前一层无法表达任务中的真实不确定性时，才值得向后一层移动。例如：

- 路径固定时，不需要 ReAct；
- 一两步能完成时，不需要 Plan；
- 没有跨会话需求时，不需要长期 Memory；
- 单一角色足够时，不需要 Multi-Agent；
- 一次请求内结束时，不需要 Autonomous Loop。

## 小结

| 问题 | 结论 |
|---|---|
| 八种模式是否必须同时使用 | 不需要，它们解决不同层次的问题 |
| 是否用了模式才算 Agent | 不是，Agent 的定义不由模式数量决定 |
| 学习重点是什么 | 意图、结构、反馈、代价和适用边界 |
| 实际设计从哪里开始 | 从最简单可行结构开始，再按问题增加模式 |

下一篇先看 Tools。它最适合帮助我们理解 Agent 系统中的一个基本分工：模型负责表达行动意图，确定性系统负责验证并执行行动。
